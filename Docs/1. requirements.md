# 직원 긴급 연락망 시스템

## 1. 비즈니스 요구사항

- 직원의 기본 연락 정보를 조회할 수 있어야 한다.
- 직원 정보를 CSV/JSON으로 업로드 및 직접 입력으로 일괄 등록할 수 있어야 한다.
- 이름으로 특정 직원을 조회할 수 있어야 한다.

## 2. 인터페이스 요구사항

### 2.1 관리 데이터

**직원 1건은 아래 정보를 가진다.**
- `id`
- 이름 (`name`)
- 이메일 (`email`)
- 핸드폰 (`tel`)
- 입사일 (`joined`)

### 2.2 공통 데이터 규칙

- 이름 중복은 허용한다.
- 이메일은 `trim` 후 소문자로 정규화하여 저장한다.
- 전화번호는 입력 시 하이픈 포함/미포함을 모두 허용하고, 저장은 숫자만 한다.
- 전화번호는 정규화 후 `010`으로 시작하는 11자리만 허용한다.
- 이메일과 전화번호는 각각 시스템 전체에서 유일해야 한다.
- 날짜 형식은 CSV/JSON 모두 `yyyy-MM-dd`만 허용한다.

### 2.3 조회 기능

### [GET] /api/employee?page={page}&pageSize={pageSize}

**기능:**
- 전체 직원 목록 조회
- 페이지 번호, 페이지 크기 기반 페이징
- 기본값은 `page=1`, `pageSize=20`
- `pageSize` 최대값은 `100`
- 정렬은 `name ASC`, 그다음 `id ASC`

**응답:**
```json
{
  "items": [
    {
      "id": "0f8fad5b-d9cb-469f-a165-70867728950e",
      "name": "김철수",
      "email": "charles@clovf.com",
      "tel": "01075312468",
      "joined": "2018-03-07"
    }
  ],
  "page": 1,
  "pageSize": 20,
  "totalCount": 1,
  "totalPages": 1
}
```

**오류 응답:**
- 잘못된 `page`, `pageSize`는 `400 Bad Request` + `ValidationProblemDetails`

### [GET] /api/employee/{name}

**기능:**
- 입력한 이름과 일치하는 직원 연락 정보 목록 조회
- 검색어는 `trim` 후 exact match 처리
- 이름 중복을 허용하므로 배열을 반환한다.
- 결과가 없으면 `200 OK`와 빈 배열 `[]`를 반환한다.
- 정렬은 `name ASC`, 그다음 `id ASC`

**응답:**
```json
[
  {
    "id": "0f8fad5b-d9cb-469f-a165-70867728950e",
    "name": "김철수",
    "email": "charles@clovf.com",
    "tel": "01075312468",
    "joined": "2018-03-07"
  }
]
```

### 2.4 등록 기능

### [POST] /api/employee

**기능:**
- 직원 정보를 일괄 등록한다.
- 파일 업로드와 직접 텍스트 입력을 모두 지원한다.
- 부분 성공을 허용한다.

**지원 Content-Type:**
- `multipart/form-data`
- `text/plain`
- `text/csv`
- `application/json`

**요청 규칙:**
- `multipart/form-data`는 파일 파트 `employeesFile`만 허용한다.
- `text/plain`은 서버가 먼저 JSON 배열 파싱을 시도하고, 실패하면 CSV로 처리한다.
- `text/csv`는 CSV로 처리한다.
- `application/json`은 JSON 배열만 허용한다.
- 지원하지 않는 `Content-Type`은 `415 Unsupported Media Type`으로 응답한다.
- 파일 파트 누락, 빈 본문, 전체 파싱 실패, JSON 배열 아님, CSV 구조 오류는 `400 Bad Request` + `ProblemDetails`로 응답한다.

**CSV 규칙:**
- 헤더 유무를 모두 허용한다.
- 헤더가 있다면 `name,email,tel,joined` 순서여야 하며, 대소문자만 무시한다.
- 큰따옴표 인용은 지원하지 않는다.
- 각 행은 정확히 4개 컬럼이어야 한다.

**CSV 데이터 예시:**
```csv
김철수,charles@clovf.com,01075312468,2018-03-07
박영희,matilda@clovf.com,01087654321,2021-04-28
홍길동,kildong.hong@clovf.com,01012345678,2015-08-15
```

**JSON 데이터 예시:**
```json
[
  { "name": "김클로", "email": "clo@clovf.com", "tel": "010-1111-2424", "joined": "2012-01-05" },
  { "name": "박마블", "email": "md@clovf.com", "tel": "010-3535-7979", "joined": "2013-07-01" },
  { "name": "홍커넥", "email": "connect@clovf.com", "tel": "010-8531-7942", "joined": "2019-12-05" }
]
```

**처리 규칙:**
- 기존 DB와 이메일/전화번호가 중복된 레코드는 실패로 처리하고, 나머지 정상 건은 저장한다.
- 같은 요청 내부에서 이메일/전화번호가 중복된 레코드도 실패로 처리하고, 나머지 정상 건은 저장한다.
- 부분 성공 시에도 모든 실패 항목을 응답에 포함한다.
- 하나라도 생성되면 `201 Created`를 반환한다.
- 한 건도 생성되지 않으면 `400 Bad Request` + `ProblemDetails`를 반환한다.

**성공 응답 예시:**
```json
{
  "total": 5,
  "created": 3,
  "failed": 2,
  "errors": [
    { "row": 2, "field": "email", "code": "duplicate_email", "message": "email already exists" },
    { "row": 4, "field": "joined", "code": "invalid_joined", "message": "joined must be yyyy-MM-dd" }
  ]
}
```

**오류 항목 규칙:**
- `errors` 항목은 `row`, `field`, `code`, `message`를 포함한다.
- `row`는 데이터 기준 1부터 시작한다.
- 기본 `code` 값은 아래를 사용한다.
  - `invalid_content_type`
  - `invalid_format`
  - `invalid_name`
  - `invalid_email`
  - `invalid_tel`
  - `invalid_joined`
  - `duplicate_email`
  - `duplicate_tel`
