# TDD 및 구현 가이드

## 1. 개발 방식

- Red: 실패하는 테스트를 먼저 작성한다.
- Green: 테스트를 통과시키는 최소 구현만 추가한다.
- Refactor: 중복 제거와 구조 개선을 수행한다.

구현 순서는 도메인 규칙 -> Application 유스케이스 -> API 통합 순서를 따른다.

## 2. 테스트 전략

### Domain Tests

- 이메일 trim + 소문자 정규화 테스트
- 전화번호 하이픈 제거 정규화 테스트
- 잘못된 전화번호 거부 테스트
- 잘못된 이메일 거부 테스트
- `yyyy-MM-dd` 이외 날짜 형식 거부 테스트

### Application Tests

- 목록 조회 핸들러 테스트
- 이름 조회 핸들러 테스트
- 이름 조회 `trim` exact match 테스트
- CSV 등록 Command 테스트
- JSON 배열 등록 Command 테스트
- `text/plain` JSON 우선 판별 테스트
- 부분 성공 응답 생성 테스트
- 중복 이메일/전화번호 실패 테스트
- 요청 내부 중복 실패 테스트
- Validator 테스트

### Integration Tests

- `GET /api/employee` 정상 조회
- `GET /api/employee` 잘못된 페이징 파라미터 검증
- `GET /api/employee/{name}` 조회 결과 없음은 빈 배열 반환
- `POST /api/employee` `multipart/form-data` CSV 등록 성공
- `POST /api/employee` `text/csv` 등록 성공
- `POST /api/employee` `application/json` 배열 등록 성공
- `POST /api/employee` `text/plain` JSON 판별 성공
- `POST /api/employee` `text/plain` CSV 판별 성공
- `POST /api/employee` 부분 성공 응답 검증
- `POST /api/employee` 지원하지 않는 Content-Type 실패
- `POST /api/employee` 잘못된 포맷 실패
- ProblemDetails 응답 검증

## 6. Definition of Done

- 모든 핵심 유스케이스 테스트가 녹색이다.
- Swagger에서 계약 확인이 가능하다.
- 실패 응답이 ProblemDetails 규격을 따른다.
- 부분 성공 등록 응답이 계약대로 반환된다.
- 최소 1개의 tracing/metrics exporter 연결 경로가 준비되어 있다.
- Docs만 보고 새 개발자가 구조를 이해할 수 있다.
