# 직원 긴급 연락망 시스템 아키텍처

## 1. 아키텍처 원칙

- 비즈니스 규칙은 프레임워크에 의존하지 않는다.
- 상위 계층은 하위 구현 세부사항을 모른다.
- 모든 유스케이스는 Application 계층을 통해 실행된다.
- 외부 입출력은 Infrastructure와 Presentation에서 담당한다.
- 테스트는 도메인과 유스케이스를 우선 보호한다.

## 2. 권장 솔루션 구조

```text
EmployeeContacts.slnx
├─ src
│  ├─ EmployeeContacts.Api
│  ├─ EmployeeContacts.Application
│  ├─ EmployeeContacts.Domain
│  └─ EmployeeContacts.Infrastructure
└─ tests
   ├─ EmployeeContacts.Domain.Tests
   ├─ EmployeeContacts.Application.Tests
   └─ EmployeeContacts.Api.IntegrationTests
```

## 3. 계층 책임

### Domain

- `Employee` 엔터티
- 이메일/전화번호 정규화 규칙
- 도메인 예외
- 순수 비즈니스 규칙

### Application

- CQRS 유스케이스
- MediatR Command/Query/Handler
- FluentValidation Validator
- DTO/응답 모델
- 저장소 인터페이스
- 트랜잭션 추상화

### Infrastructure

- EF Core `DbContext`
- Entity 매핑
- Repository 구현
- 마이그레이션
- CSV/JSON parser 구현
- raw text 포맷 판별 구현
- OpenTelemetry/로깅 실제 연동

### Api

- Controller 또는 Minimal API 엔드포인트
- ProblemDetails 응답 매핑
- Swagger 설정
- DI 조립
- 인증/인가가 생길 경우 진입점 계층에서 처리

## 4. 프로젝트 참조 방향

```text
Api -> Application
Api -> Infrastructure
Infrastructure -> Application
Infrastructure -> Domain
Application -> Domain
Domain -> (없음)
```

- `Application`은 `Infrastructure`를 참조하지 않는다.
- `Domain`은 어떤 프로젝트도 참조하지 않는다.

## 7. 데이터 모델

### Employee Aggregate

- `Id : Guid`
- `Name : EmployeeName`
- `Email : EmployeeEmail`
- `PhoneNumber : EmployeePhoneNumber`
- `Joined : DateOnly`
- `CreatedAt : DateTimeOffset`
- `UpdatedAt : DateTimeOffset`

설명:
- Domain은 값 객체를 공개 계약으로 사용한다.
- API response body에서는 `PhoneNumber`를 외부 계약에 맞춰 `tel`로 매핑한다.

### 인덱스

- `Email` 유니크 인덱스
- `PhoneNumber` 유니크 인덱스
- `Name` 조회 인덱스

## 8. EF Core 설계

- 초기 구현 DB는 SQLite로 고정한다.
- `EmployeeConfiguration`에서 길이/필수/인덱스를 명시한다.
- 날짜는 DB provider에 맞는 형식으로 매핑한다.
- 조회는 `AsNoTracking`을 기본으로 사용한다.
- 정렬 기본값은 `Name`, 그다음 `Id` 오름차순이다.

## 9. 예외 및 ProblemDetails

### 예외 분류

- Validation 실패
- 리소스 조회 결과 없음
- 중복 데이터
- 파싱 실패
- 예기치 않은 서버 오류

### 응답 원칙

- 잘못된 쿼리 파라미터는 `400 Bad Request` + `ValidationProblemDetails`
- 지원하지 않는 `Content-Type`은 `415 Unsupported Media Type`
- 요청 자체 오류는 `400 Bad Request` + `ProblemDetails`
- 등록 요청에서 한 건도 생성되지 않으면 `400 Bad Request` + `ProblemDetails`
- 모든 예외는 표준 status code와 `type`, `title`, `detail`, `traceId`를 포함
- 내부 예외 메시지는 외부에 직접 노출하지 않는다

## 10. 관측성 설계

### Logging

- 요청 시작/종료 로그
- Command/Query 처리 로그
- 예외 로그
- 대량 등록 건수 로그

### OpenTelemetry Tracing

- ASP.NET Core incoming request
- MediatR pipeline span
- EF Core DB span
- 필요 시 parser span 추가

### OpenTelemetry Metrics

- HTTP request duration/count
- 등록 성공 건수/실패 건수
- 파싱 실패 건수

## 11. Swagger 설계

- `/swagger`에서 API 확인 가능
- 요청/응답 예제 제공
- CSV text 입력 예시, JSON 입력 예시 명시
- `multipart/form-data` 업로드 예시에서 `employeesFile` 필드 사용
- ProblemDetails 스키마 노출
- 부분 성공 응답(`total`, `created`, `failed`, `errors`) 예시 제공

## 12. 확장 포인트

- 파일 스토리지 기반 업로드
- 비동기 배치 등록
- 이름 검색을 exact match에서 contains 검색으로 확장
- 인증/인가 추가
- 멀티 테넌시 또는 부서 기준 분리
