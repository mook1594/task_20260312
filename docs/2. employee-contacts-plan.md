# 직원 긴급 연락망 시스템 개발 계획

## 1. 목표

- 요구사항 문서의 직원 조회/등록 API를 .NET 10 기반으로 구현한다.
- 유지보수성과 테스트 가능성을 위해 클린 아키텍처를 적용한다.
- 기능 구현은 TDD 흐름으로 진행한다.
- 요청/응답 처리 흐름은 CQRS + MediatR + FluentValidation 조합으로 표준화한다.
- 운영 품질을 위해 ProblemDetails, 로깅, OpenTelemetry, Swagger를 기본 포함한다.

## 2. 범위

### 포함

- 직원 목록 조회 API
- 이름 기준 직원 목록 조회 API
- CSV/JSON 기반 직원 일괄 등록 API
- 직접 입력 텍스트 기반 CSV/JSON 등록 API
- 데이터 영속화
- 예외 응답 표준화
- API 문서화
- 추적/메트릭/로그 관측성
- 자동화 테스트

### 제외

- 인증/인가
- 프론트엔드 UI
- 대용량 배치 튜닝
- 다국가 전화번호/날짜 형식 지원

## 3. 핵심 결정사항

- 프로젝트 런타임은 `C# / .NET 10`으로 고정한다.
- 개발 DB는 `SQLite`로 고정한다.
- API 진입점은 ASP.NET Core Web API 단일 호스트로 시작한다.
- 내부 구조는 `Presentation`, `Application`, `Domain`, `Infrastructure` 계층으로 분리한다.
- 읽기/쓰기 유스케이스는 각각 Query/Command로 분리한다.
- 모든 Command/Query는 MediatR를 통해 처리한다.
- 입력 검증은 FluentValidation 파이프라인으로 일원화한다.
- 데이터 접근은 EF Core를 사용한다.
- 예외는 ProblemDetails 형태로 일관되게 응답한다.
- OpenTelemetry로 tracing/metrics를 수집한다.
- Swagger(OpenAPI)로 계약을 노출한다.

## 4. 기능별 구현 계획

### 4.1 직원 목록 조회

- `GET /api/employee?page={page}&pageSize={pageSize}`
- 이름 오름차순, 그다음 `Id` 오름차순 정렬
- 기본 페이징 값은 `page=1`, `pageSize=20`
- `pageSize` 최대값은 `100`
- 결과는 `items`, `page`, `pageSize`, `totalCount`, `totalPages`를 반환
- 잘못된 쿼리 파라미터는 `400 Bad Request` + `ValidationProblemDetails`

### 4.2 직원 이름 조회

- `GET /api/employee/{name}`
- 경로 파라미터 방식을 유지한다.
- 입력 이름은 `trim` 후 exact match 조회
- 이름 중복 허용 정책이므로 배열 반환
- 결과 없음은 `200 OK`와 빈 배열 반환
- 결과 정렬은 이름 오름차순, 그다음 `Id` 오름차순
- 응답 모델에는 `Id`를 포함

### 4.3 직원 일괄 등록

- `POST /api/employee`
- 지원 Content-Type
  - `multipart/form-data`
  - `text/plain`
  - `text/csv`
  - `application/json`
- 입력 원천
  - `multipart/form-data`의 `employeesFile`
  - 요청 본문의 raw text 또는 raw payload
- 입력 포맷
  - CSV
  - JSON 배열
- `text/plain`은 JSON 파싱 먼저 시도 후 실패하면 CSV로 처리
- 파싱 후 공통 DTO로 정규화하여 동일한 Command로 처리
- 부분 성공 허용
- 기존 데이터 중복과 요청 내부 중복은 실패 목록으로 반환
- 하나라도 생성되면 `201 Created`
- 일괄 등록 응답에는 `Location` 헤더를 포함하지 않는다
- 한 건도 생성되지 않으면 `400 Bad Request` + `ProblemDetails`
- 지원하지 않는 Content-Type은 `415 Unsupported Media Type`

## 5. 데이터 및 도메인 규칙

- 직원 엔터티 필드: `Id`, `Name`, `Email`, `Tel`, `Joined`
- 이름 중복 허용
- 이메일은 trim 후 소문자로 정규화하여 저장
- 전화번호는 하이픈 포함/미포함 입력을 허용하고 숫자만 저장
- 전화번호는 `010` 시작 11자리만 허용
- 이메일/전화번호는 시스템 전체에서 유일해야 한다
- 날짜는 저장 시 `DateOnly`를 우선 적용한다
- CSV/JSON 날짜는 모두 `yyyy-MM-dd`만 허용한다
- CSV 오류 응답의 `row`는 헤더 존재 여부와 무관하게 실제 데이터 행 기준 1부터 시작한다
